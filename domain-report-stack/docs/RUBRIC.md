# 缺陷分级与评分（domain-report-stack）

## P0 — 阻断

- 事实错误、不可接受的**领域串台**（行业/项目/地域明显错配）。
- 关键**法规或政策引用错误**且影响结论。
- 结论与证据链**明显矛盾**且未标注为假设。

**规则**：存在**未关闭的 P0** → **不得对外宣称定稿**；须先修复或撤回结论。

## P1 — 严重

- **核心判断缺依据**（无来源、无测算、无逻辑链）。
- **规定性结构缺失**（该类型报告通常必有的章/表/假设说明缺失）。
- **重大风险未披露**（足以改变读者决策的 downside）。

**规则**：本轮修复周期内应处理；未处理须在文首「未决事项」列出并**降级结论强度**。

## P2 — 重要

- 论证偏薄、**口径混用**、章节组织影响理解、建议与正文不一致。

**规则**：纳入修复计划；可分批次；若交付「客户版」且客户在意深度，应处理后再发。

## P3 — 一般

- 表达、层级、非关键呈现问题；**不改变实质判断**。

**规则**：可延后；可交给版式/文笔或导出层（P4）。

## P4 — 润色（可选）

- 版式、视觉、模板美化；**不参与合格判定**。

## 边界：缺证据

- **找不到证据 ≠ 自动 P0**。
- 缺**关键**数据却写**确定结论** → 倾向 **P0/P1**（越权断言）。
- 仅**次要细节**缺失 → **P2**，改为「待补充」或弱化表述。

## 评分

- 使用**多维 rubric**（如：事实与引用、逻辑链、完整性、商业可读性、风险披露）+ 总分。
- **硬规则**：任一 **P0 未关闭** → 总分视为**不合格档**（建议总分封顶 **59**，具体分数线可由项目约定）。

## 执行摘要 vs 缺陷表

- **缺陷表**：枚举**全部**问题，**条数不限**；每条含：id、级别、位置、描述、证据/依据、建议改法。
- **执行摘要**：建议**结论**与**风险**各 **3–5 条**（可配置），并写明「详见缺陷表」；**不得**将「3–5 条」理解为全案只允许少量错误。

## remediation 蓝图对齐

研读旧稿或新稿二轮提质时，须产出：文档类型判定 → 执行摘要 → 完整缺陷表 → 多维评分 → 修订计划 → 修订稿/变更说明（见 `pipelines/blueprints/...yaml` 中 `remediation` 章节列表）。
